# 工作跟踪指南

本文档说明如何使用 RFC 来跟踪项目工作。

## 原则

**所有重要的工作都应该用 RFC 记录**，包括但不限于：

1. **问题修复**
   - 构建错误修复
   - 路径问题修复
   - 配置问题修复

2. **功能开发**
   - 新功能实现
   - 功能改进
   - 性能优化

3. **架构变更**
   - 项目结构重组
   - 依赖关系变更
   - 技术栈升级

4. **流程改进**
   - 构建流程改进
   - 开发流程改进
   - 测试流程改进

## RFC 创建流程

### 1. 识别工作

当开始一项重要工作时，应该：

1. **评估工作重要性**
   - 是否影响多个文件/项目？
   - 是否改变项目结构？
   - 是否影响其他开发者？
   - 是否需要后续参考？

2. **如果重要，创建 RFC**
   - 使用模板创建新的 RFC
   - 记录问题、解决方案、实施细节
   - 更新 RFC README

### 2. RFC 编号

- 按时间顺序编号
- 编号一旦分配，不再重用
- 即使工作被取消，编号也保留

### 3. RFC 状态

RFC 状态应该及时更新：

- **提案 (Proposed)** - 刚创建，待评审
- **进行中 (In Progress)** - 正在实施
- **已完成 (Completed)** - 工作已完成
- **已拒绝 (Rejected)** - 工作被取消
- **已废弃 (Deprecated)** - 被更好的方案替代

## 示例

### 示例 1: 构建错误修复

**场景**: 发现构建错误，需要修复

**步骤**:
1. 创建 RFC-XXXX: 构建错误修复
2. 记录问题描述、根本原因、解决方案
3. 实施修复
4. 更新 RFC 状态为"已完成"
5. 记录验证结果

### 示例 2: 路径问题修复

**场景**: 发现多个项目文件路径不一致

**步骤**:
1. 创建 RFC-XXXX: 路径问题修复
2. 记录问题范围、修复方案
3. 创建修复脚本
4. 执行修复
5. 更新 RFC 状态为"已完成"
6. 记录修复统计

## 最佳实践

1. **及时记录**
   - 不要等到工作完成才记录
   - 在开始工作时就创建 RFC

2. **详细记录**
   - 记录问题、原因、解决方案
   - 记录实施细节和验证方法
   - 记录影响和后续工作

3. **保持更新**
   - 工作进展时更新 RFC
   - 完成后更新状态
   - 记录变更历史

4. **相互引用**
   - 相关 RFC 之间相互引用
   - 在 README 中保持列表更新
